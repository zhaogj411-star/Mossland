# 代理操作指南

本仓库使用仓库内记忆。Codex 会话、协作者和未来 worktree 必须从本仓库文件恢复项目上下文，不依赖私有聊天历史、本地 shell 历史、IDE 状态或个人 Codex 配置。

## 启动

每次 Codex 会话进入本仓库时：

1. 读取 `AGENTS.md`。
2. 如果存在，读取 `docs/agent-start.md`。
3. 如果存在，读取 `docs/memory/current.md`。
4. 如果存在，读取 `docs/README.md`。
5. 如果任务涉及代码且存在 `docs/code-index.md`，读取它。
6. 只有当当前胶囊指向、任务需要历史细节或任务状态不清楚时，才读取 `docs/memory/progress.md` 或 `docs/memory/inbox.md`。
7. 只读取与当前任务相关的其他文档。
8. 如果项目还没有实现代码，把它当作空代码库，并根据仓库文档判断下一步要构建什么。

## 多代理协作

- 对于可以安全拆分的较大工作，Codex 应优先使用多个代理或子代理并行调查、实现和验证。
- 委派并行工作前，协调会话必须为每个可能写文件的代理分配清晰的文件或目录所有权。
- 不同代理不能同时修改同一个文件。如果多个工作流需要同一文件，应通过单一 owner 串行编辑，让其他代理提供 notes 或 patch 建议，或由协调者在并发写入结束后合并。
- 允许只读重叠；不允许写入重叠。
- 协调者仍负责复查最终 diff、解决集成问题、运行合适检查，并更新仓库记忆。

## 代码设计原则

- 保持代码简单：优先选择直接、可读的实现，不做聪明或猜测性的抽象。
- 保持模块聚焦：每个模块应拥有清晰职责和狭窄公共接口。
- 在新增模式前，先复用已有模式、helper 和配置路径；只有当重复变得有意义时才抽取共享逻辑。
- 保持单个文件小而易扫读。当文件职责过宽、混杂无关内容或让未来修改难以定位时，拆分文件。

## 文档与记忆

- `docs/` 是持久项目记忆根目录。
- 文档首先服务于 Codex 的持久上下文恢复。可读性很重要，但文档的首要目标是让未来 Codex 会话无需私有聊天历史即可恢复意图、架构、约束、决策和下一步动作。
- 文档保持简洁。优先记录规则、决策、交接说明、索引和可复用命令知识，避免长篇叙事历史。
- 索引是强制要求。新的持久文档必须能从 `docs/README.md`、`docs/code-index.md` 或同一文档子树中命名清楚的本地索引找到。
- 索引条目应说明 Codex 何时需要读取该文档，而不是只列文件名。
- 本仓库不固定文档分类。Codex 可根据项目形态在 `docs/` 下创建目录和文件。
- 优先使用清晰名称，不强求僵硬结构。
- `docs/memory/inbox.md` 用于未归类记录、临时发现，以及需要后续人工或 Codex 分拣的 hook。
- `docs/memory/current.md` 用于新 Codex 会话应立即恢复的短上下文。
- `docs/memory/progress.md` 用于比 `current.md` 更需要历史的当前工作状态、下一步和阻塞项。
- `docs/code-index.md` 用作包、入口点、配置、测试和外部资源边界的紧凑地图。
- `docs/agent-start.md` 和 `docs/agent/commands.md` 用于避免重复启动研究和命令猜测。
- 当稳定知识的目标位置清楚时，把它从收件箱移到命名更好的文档。
- 保存记忆时，判断应更新现有文档、新建文件还是新建目录，并在同一次变更中清理过期或重复记忆。
- 遵循 `docs/memory/maintenance.md` 进行记忆放置和清理。
- 工作期间保持 `docs/memory/current.md` 最新，尤其是在压缩、停止或交接给另一个会话前。把较长历史移到 `docs/memory/progress.md`。
- 新增、移动、重命名代码或集成其他项目时，保持 `docs/code-index.md` 最新。
- 新增、移动、重命名或废弃持久文档时，保持 `docs/README.md` 最新。
- README 只应存在于有导航价值或模块边界的位置。优先放在根目录、主要包、工作流、脚本、测试和 docs 索引处。不要在每个实现或资源子目录添加很薄的 README，如果父 README 能清楚解释即可。

## Codex 必须记录什么

当 Codex 学到或创建有用的持久信息时，必须写入 `docs/`，包括：

- 产品需求、用户意图、工作流和约束。
- 设计决策及其权衡。
- 架构、数据格式、配置语义和公共接口。
- 设置、运行、测试、部署和故障排查流程。
- 外部参考资料，用 Codex 自己的话总结并附链接。
- 工作暂停或完成时的交接记录。

不要把 secrets、私有凭据、访问 token 或个人机器状态存入仓库记忆。

## 对话期间

当用户说 “remember”、“record this”、“note this”、“ji yi xia”、“记一下”、“记录”、“以后记住” 或 “下次注意” 时，立即把简洁记录写入 `docs/memory/inbox.md`，或写入 `docs/` 下更合适的文件。

当讨论产生设计、需求或决策时，在同一轮内更新 `docs/`，不要等到提交时再做。

进行代码变更时，如果变更影响行为、设置、架构、接口或未来工作，结束前更新或创建相关文档。

当文档过期时，重写、移动、合并、标记为已废弃或删除过期记忆，避免未来会话恢复错误上下文。

## Hook 规则

本仓库拥有 `.codex/hooks.json`，用于把 Codex 生命周期事件连接到 `agent-code/scripts/codex-hook.sh`。

Hook 是提醒器和上下文注入器，不能替代 Codex 判断：Codex 仍必须决定哪些信息是持久信息，并写入正确文档。共享文件系统上应保持高频 `PreToolUse` / `PostToolUse` hook 禁用；需要显式验证时，使用 `agent-code/scripts/agent/impact.sh` 和 `agent-code/scripts/agent/check.sh`。

## 代理 Harness 语言约定

面向人阅读的文档、agent harness 说明、hook 提醒、脚本输出和未来 harness 设定默认使用中文。命令名、路径、scope、事件名、JSON key、代码标识和外部接口名按原接口保留英文，避免破坏配置或自动化。

## 项目状态

Mossland 当前是轻量仓库。直到实现代码、资源和工作流出现前，保持代理基础设施通用。不要从其他项目复制长篇文档，除非它描述的是当前 Mossland 规则、决策或可复用工作流。
